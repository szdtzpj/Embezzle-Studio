# Provider Protocol Matrix

Last verified: 2026-07-12. This file records wire-level decisions that must stay covered by tests when model catalogs change. Embezzle Studio operates no provider proxy or billing service: every request uses a user-configured endpoint/key and every provider charge belongs to that user's account.

## Provider setup and capability evidence

- Provider kind, canonical Base URL, and key form one security binding. The setup wizard edits them in an independent draft. If kind or endpoint changes, saving clears the previous key, models, and discovery candidates before any request can use the new destination; the old key is never probed against the new endpoint.
- Exact official OpenAI, Volcengine Ark, and Alibaba Bailian hosts are classified locally. Lookalike hosts do not gain official-provider routing. New API and custom OpenAI-compatible endpoints remain best-effort compatibility profiles and are not promoted to an official adapter by their display name.
- Bailian pay-as-you-go compatible endpoints are distinct from Coding Plan/Token Plan subscription endpoints and `sk-sp-` credentials, which are blocked for this custom application. Changing to a valid pay-as-you-go endpoint requires a fresh key binding. [Bailian tools and plan restrictions](https://help.aliyun.com/zh/model-studio/more-tools)
- The capability matrix keeps provider/model-declared support separate from client-supported serializers/parsers. Discovery metadata can explain what a provider advertises but cannot enable a route, attachment type, hosted tool, or parameter that this client has not implemented and tested.

## OpenAI

- Ordinary chat models use `POST /v1/chat/completions`. The request uses SSE plus `stream_options.include_usage`; the final usage event can have an empty `choices` array. [Chat Completions reference](https://developers.openai.com/api/reference/resources/chat/subresources/completions/methods/create)
- `gpt-5-pro`, versioned GPT-5.x Pro models, `o1-pro`, and `o3-pro` are routed to `POST /v1/responses`. The app sends `input` and nested `reasoning.effort`; it does not send Chat-only `messages` or sampling fields. [Responses migration guide](https://developers.openai.com/api/docs/guides/migrate-to-responses)
- When the local output cap is enabled, official Chat sends `max_completion_tokens`; Responses (including provider-hosted search) sends `max_output_tokens`. The cap bounds the documented text-output field but is not a provider bill ceiling. [Chat Completions reference](https://developers.openai.com/api/reference/resources/chat/subresources/completions/methods/create), [Reasoning token allocation](https://developers.openai.com/api/docs/guides/reasoning#allocating-space-for-reasoning)
- Effort values remain distinct in storage and UI: GPT-5 uses `minimal`; GPT-5.1 uses `none`; GPT-5.2+ adds `xhigh`; GPT-5.6 adds `max`; Pro model IDs use their documented subset. GPT-5.6 pro mode is a Responses `reasoning.mode`, not a separate `gpt-5.6-pro` model slug. [Reasoning guide](https://developers.openai.com/api/docs/guides/reasoning), [GPT-5](https://developers.openai.com/api/docs/models/gpt-5), [GPT-5.2](https://developers.openai.com/api/docs/models/gpt-5.2), [GPT-5.6 guidance](https://developers.openai.com/api/docs/guides/latest-model)
- Active reasoning omits sampling parameters. When sampling is supported, the client sends at most one of `temperature` and `top_p`. GPT Image omits `response_format`; DALL-E requests `b64_json` so expiring URLs are not persisted. [Latest-model parameter compatibility](https://developers.openai.com/api/docs/guides/latest-model), [Images API](https://developers.openai.com/api/reference/resources/images/methods/generate)

## Volcengine Ark

- Data-plane calls use the exact official `ark.cn-beijing.volces.com`/`ark.cn-beijing.volcengineapi.com` host with Bearer API Key. Display names and lookalike hostnames never activate the Ark protocol. Account Endpoint discovery remains an AK/SK control-plane operation that the mobile app does not call. [Authentication and Base URL](https://www.volcengine.com/docs/82379/1298459?lang=zh), [Ark API reference](https://api.volcengine.com/api-docs/?serviceCode=ark&version=2024-01-01)
- Ark's public model-list page is a catalog, not a documented Bearer data-plane `GET /api/v3/models` contract. The exact official host is allowed a best-effort compatibility probe because it returned structured model metadata in a 2026-07-10 live smoke test; any error, empty result, or incompatible result falls back to curated local candidates maintained from the official catalog. Shutdown/unsupported tasks are filtered, retiring entries are labelled, and this probe is never required for manual Endpoint IDs. [Official model catalog](https://www.volcengine.com/docs/82379/1330310), [OpenAI SDK compatibility](https://www.volcengine.com/docs/82379/1330626)
- The app's Ark conversation route uses Chat: `/chat/completions`, `messages`, top-level `thinking`, and top-level `reasoning_effort`. GLM 5.2 can express `none` through `max`; DeepSeek V4 preserves native `max`; Doubao Seed 2.x uses `minimal` through `high`; toggle-only families expose only on/off. Ark Responses is a distinct `/responses` protocol with `input`, top-level `thinking`, and nested `reasoning.effort`; the app does not silently mix the two shapes. [Chat API](https://www.volcengine.com/docs/82379/1494384), [Chat deep thinking](https://www.volcengine.com/docs/82379/1449737#fc5eac89), [Responses API](https://www.volcengine.com/docs/82379/1569618), [Responses deep thinking](https://www.volcengine.com/docs/82379/1956279#dc4c1547)
- The local output cap maps to `max_tokens` for ordinary Ark Chat and to `max_output_tokens` for Ark Responses/search. It is omitted for image/video generation tasks. [Ark Chat API explorer](https://api.volcengine.com/api-docs/view?action=ChatCompletions&serviceCode=ark&version=2024-01-01)
- Discovered capabilities describe adapters that are actually wired, not every capability advertised by the provider: Ark Chat currently accepts image input but does not expose video input; image generation is text-to-image only; video-generation tasks retain supported reference image/video inputs. Unsupported 3D and pure speech-to-text entries are not offered as chat models.
- Video tasks use `/contents/generations/tasks`; polling treats `queued`, `running`, `cancelled`, `succeeded`, `failed`, and `expired` distinctly. Reference images/videos are forwarded instead of silently dropped. [Create task](https://www.volcengine.com/docs/82379/1520757?lang=zh), [Query task](https://www.volcengine.com/docs/82379/1521309?lang=zh)

## Alibaba Bailian

- Compatible chat uses `/compatible-mode/v1/chat/completions`, including `reasoning_content`, final usage chunks, `image_url`, and provider-specific `video_url`. This remains the broadest common route. Bailian Responses is an optional Qwen-only protocol with `input` and nested `reasoning.effort`; the app does not auto-route third-party families to it by model name. [OpenAI-compatible Chat](https://help.aliyun.com/zh/model-studio/qwen-api-via-openai-chat-completions), [OpenAI-compatible Responses](https://help.aliyun.com/zh/model-studio/qwen-api-via-openai-responses)
- The local output cap maps to broad-compatible Chat `max_tokens` and Responses/search `max_output_tokens`. It does not use `max_completion_tokens` for all Bailian Chat families because that field is model-family-limited. Image/video generation omits the text cap. [Model usage statistics](https://help.aliyun.com/zh/model-studio/model-usage-statistics), [OpenAI-compatible Responses](https://help.aliyun.com/zh/model-studio/compatibility-with-openai-responses-api)
- `default` always omits thinking controls. Thinking-only QwQ/QVQ/DeepSeek R1/Qwen/Kimi/MiniMax families never receive an off field. Mixed Qwen, DeepSeek V3, Alibaba-hosted Kimi, and Stepfun use `enable_thinking`; MiniMax/MiniMax-M3 default is left wire-neutral (provider `adaptive`) and explicit off uses `thinking.type=disabled`; native GLM/DeepSeek V4/Stepfun effort subsets stay model-specific. [Deep thinking models](https://help.aliyun.com/zh/model-studio/deep-thinking/), [Kimi](https://help.aliyun.com/zh/model-studio/kimi-api), [Moonshot Kimi](https://help.aliyun.com/zh/model-studio/kimi-api-by-moonshot-ai), [MiniMax](https://help.aliyun.com/zh/model-studio/minimax-api-by-minimax), [Stepfun](https://help.aliyun.com/zh/model-studio/stepfun)
- Numeric `thinking_budget` is sent only to documented budget-capable families. The UI labels the four conservative presets as 1K/4K/8K/16K token caps instead of pretending that 16K is the provider's native “maximum.” Moonshot `kimi/...` and namespaced `MiniMax/...` direct-supplier families do not expose editable sampling fields. Alibaba-hosted Kimi and MiniMax aliases retain their documented temperature/top-p/presence fields while omitting undocumented frequency penalty; Stepfun retains sampling during thinking and clamps frequency penalty to 0..1. UI bounds and the final serializer share the same provider/model constraints. [Visual reasoning](https://help.aliyun.com/zh/model-studio/visual-reasoning), [Alibaba-hosted MiniMax](https://help.aliyun.com/zh/model-studio/minimax-api)
- GLM 5/5.1 preserve `xhigh` and never send `max`; GLM 5.2 may send `max`. DeepSeek V4 uses its native `high|max` subset. [GLM models](https://help.aliyun.com/zh/model-studio/glm), [DeepSeek models](https://help.aliyun.com/zh/model-studio/deepseek-api)
- Bailian `/models` is treated as an optional compatibility feature. A missing or malformed response directs the user to the official model catalog and manual Model ID entry instead of making the provider unusable. [Model catalog](https://help.aliyun.com/zh/model-studio/models)

## Provider-hosted search

- OpenAI search uses official `POST /v1/responses` with `tools: [{ type: "web_search" }]`. Ark uses official `POST /api/v3/responses` with bounded `max_keyword: 3` and `limit: 10`. Bailian uses official compatible `POST /responses` with the `web_search` tool. The optional text cap is `max_output_tokens` for all three Responses routes. Provider kind, official HTTPS host, and known base path must all match; a lookalike host or relay is not promoted to a hosted-search adapter. The Bailian Beijing, Singapore, and US pay-as-you-go hosts are recognized explicitly rather than through suffix matching. [OpenAI web search](https://developers.openai.com/api/docs/guides/tools-web-search), [Ark web search](https://www.volcengine.com/docs/82379/1756990), [Bailian web search](https://help.aliyun.com/zh/model-studio/web-search)
- Ark/Bailian search starts text-only. OpenAI may reuse its already validated Responses image/file input. A model capability label alone never sends a search: the user key, exact adapter, and explicit search switch are also required.
- `webSearchTriggered` requires a `web_search_call`, a valid `url_citation`, or (Bailian only) positive `usage.x_tools.web_search.count`. Citations reject non-HTTPS, credential-bearing, local/private, malformed, or out-of-range URLs and are displayed as visible links.

## Request-based audio

- Audio is Android-only and requires the user's own key. OpenAI transcription uses multipart `POST /v1/audio/transcriptions` (25 MiB maximum); synthesis uses JSON `POST /v1/audio/speech` (4,096 code-point input maximum) and stores returned bytes in the app cache. [OpenAI STT](https://developers.openai.com/api/docs/guides/speech-to-text), [OpenAI TTS](https://developers.openai.com/api/docs/guides/text-to-speech)
- Bailian transcription uses `qwen3-asr-flash` through compatible Chat `input_audio` with a 10 MiB data-URL maximum. Bailian synthesis uses provider-native multimodal generation and immediately downloads the temporary result URL after strict HTTPS/public-host validation. [Qwen ASR](https://help.aliyun.com/zh/model-studio/qwen-asr-api-reference), [Qwen TTS](https://help.aliyun.com/en/model-studio/qwen-tts-api)
- Ark credentials are never reused for Volcengine speech products, which have separate AppID/access-token/resource contracts. OpenAI Realtime is not enabled because a safe client session needs a server-generated ephemeral token.

## Local cost guard semantics

- Output-token and unknown-cost policies run before a covered request is sent. Multi-model comparison is bounded locally and can require one explicit confirmation because each selected target is an independent provider call.
- Daily request limits use the device-local attempt ledger. Daily CNY and USD thresholds compare only the already completed, locally known subtotal and gate the next attempt after that subtotal reaches the configured threshold. The client does not project the current request's cost and cannot promise that a request will not cross a provider budget.
- CNY and USD remain separate; Embezzle Studio performs no exchange-rate conversion. Search-tool charges, voice/media pricing, retries, server-side work, and missing usage can remain unknown, and unknown is never rewritten as zero.
- `providerUsageEvents` is a local attempt/estimate ledger, not an invoice or provider-authoritative usage record. It is pruned locally, excluded from encrypted/plain backup exports, and preserved from the receiving device when another workspace is imported.
- New API/custom OpenAI-compatible Chat receives best-effort `max_tokens`; the UI and documentation do not claim every relay honors it. Media generation does not receive a text output cap.

## Remote MCP

- The executable v1.4 route is restricted to the dedicated `openai-compatible` provider kind whose inspected canonical base is exactly `https://api.openai.com/v1`, producing the official `/responses` adapter. Its MCP tool definition always uses a non-empty exact `allowed_tools` array and `require_approval: "always"`; every request uses `store: false`, `parallel_tool_calls: false`, requests `reasoning.encrypted_content`, and rejects HTTP redirects. A lookalike host, custom port, unexpected path, relay/custom provider kind, Ark profile, or Bailian profile cannot enter this runtime. The local Web development proxy also owns a non-configurable `redirect: "error"` upstream policy so credentials are never forwarded to a redirect target. [OpenAI MCP](https://developers.openai.com/api/docs/guides/tools-connectors-mcp)
- Continuations do not use `previous_response_id`. The client manually carries the original input, all prior output items (including `mcp_list_tools`, reasoning, and approval items), and every `mcp_approval_response` into the next `store: false` request. The provider model, MCP definition, allowlist, and locally reinjected authorization remain bound across rounds.
- Stateless replay fails closed if any reasoning item lacks usable `encrypted_content`, if an `mcp_list_tools` item reports an error (including an ambiguous response that also supplies tools), or if any response reuses a prior `mcp_call.id`, approval ID, or response ID.
- Each `mcp_approval_request` must match the configured server label, an exact allowlisted tool that appeared in `mcp_list_tools`, a unique approval ID, and bounded valid JSON-object arguments. The full arguments are shown before the user chooses approve, deny, or cancel. A returned `mcp_call` must correlate to the approved ID, tool, and exact raw arguments; malformed, oversized, replayed, ambiguous, unknown, or parallel approval/call evidence fails closed. No remembered approval exists, and one turn permits at most four approvals.
- Approve and deny both create a new provider Response; cancel creates no continuation and aborts the active network lifecycle. The app registers each initial or continuation Response before send as a potentially billable `providerRequestCount` attempt rather than treating one chat turn as one request, and aggregates bounded usage across completed rounds. An abort may conservatively leave a pre-send attempt in the ledger, so this is not a claim about provider receipt and never replaces the provider bill.
- Safe MCP is mutually exclusive with provider-hosted web search and multi-model comparison in v1.4. Media generation, audio, embeddings, reranking, and other specialized tasks do not enter the MCP route.
- Ark documents `require_approval`, an object-shaped `allowed_tools.tool_names`, approval items, and `previous_response_id`, but the runtime remains disabled until a real Ark account proves a complete no-storage continuation from locally replayed context with `store: false`. The builder models the protocol-specific allowlist shape for future fixture work; that does not enable execution. [Ark MCP](https://www.volcengine.com/docs/82379/1827534)
- Bailian Responses documents hosted MCP calls but not an equivalent pre-execution approval pause/response contract. Adding an ignored OpenAI field or asking after execution would not create safety, so Bailian MCP remains configuration-only and non-executing. [Bailian Responses MCP](https://help.aliyun.com/en/model-studio/qwen-api-via-openai-responses), [Bailian MCP](https://help.aliyun.com/en/model-studio/mcp)
- MCP remains BYOK. Embezzle Studio operates no production API, MCP gateway, approval server, or tool worker. Provider keys and MCP authorization stay in platform-scoped secret storage; authorization is injected only into the outgoing tool definition and is excluded from chat results, receipts, errors, diagnostics, and plaintext or encrypted backups. Full tool arguments and outputs are likewise excluded from receipts and diagnostic logs. `store: false` controls Responses object storage only and does not override provider security logs, organization data controls, or remote-MCP retention. Real OpenAI-account plus trusted-MCP-server Android acceptance is still an external boundary, not a result inferred from protocol tests. See [v1.4 Safe MCP Tools](./safe-mcp-tools.md) for the complete trust and release boundary.
- Importing an encrypted workspace never re-enables a remote MCP entry, even when the exported configuration recorded `enabled: true`; every imported server requires a fresh local review and enable action. Internal storage likewise forces non-official provider bindings (Ark, Bailian, custom, or malformed OpenAI routes) to `enabled: false` when saving or loading.
